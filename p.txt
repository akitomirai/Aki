1. 业务流程与角色需求（对应研究内容 1）
1.1 全链路业务流程（田间到餐桌）

生产：基地/地块建档 → 种植批次创建 → 日常农事记录（施肥/用药/浇灌/采收）

加工：入厂登记 → 加工环节记录（清洗/分级/包装/批次拆分/合批可选）

检验：送检 → 质检报告上传/录入 → 生成报告哈希指纹并存证

仓储：入库/出库 → 库位/温湿度（可对接传感器）

物流：发货 → 运输节点更新（轨迹、承运方、到达签收）

销售：上架 → 终端售卖信息（门店/平台/时间）

消费者查询：扫码 → 查询履历展示 → 记录查询行为（次数/时间/IP/地理位置）

1.2 核心角色与差异化需求

生产者/企业管理员：创建批次、维护基础档案、授权人员、导出数据、生成二维码

环节操作员（加工/仓储/物流/门店）：按权限录入节点信息、上传附件（图片/报告）

监管人员：查看全链路、抽检记录、异常批次、追责溯源、查询统计看板

消费者：无需登录，扫码即查；信息直观可信；可看“查询热度/最近查询时间”等

3. 核心数据设计：MySQL + Redis + 哈希存证（对应研究内容 3.1）
3.1 数据建模思路（“批次为主线，节点为事件”）

批次（Batch）：贯穿全链路的主实体

溯源节点（TraceEvent）：每个环节的一条事件记录（时间、地点、操作者、内容、附件）

核心可信数据（Quality/Pesticide 等）：仍存 MySQL 明细，同时生成 哈希指纹 存证表，做到“可校验”

3.2 MySQL 表设计（毕业设计级别够用，字段可按需精简）

下面是建议的最小可用表（MVP）：

1）用户与权限

sys_user(id, username, password, real_name, phone, status, org_id, created_at, updated_at)

sys_role(id, code, name)

sys_user_role(user_id, role_id)

sys_perm(id, code, name)（可选，简单项目可用“角色->菜单”替代）

sys_role_perm(role_id, perm_id)（可选）

2）基础档案

org_company(id, name, license_no, address, contact)

base_product(id, name, category, spec, unit)（如：赣南脐橙）

base_farm(id, company_id, name, location, area)（基地）

base_plot(id, farm_id, name, geo_lat, geo_lng)（地块，可选）

3）批次与节点（核心）

trace_batch(id, batch_code, product_id, company_id, farm_id, plot_id, start_date, status, remark)

trace_event(id, batch_id, stage, event_time, operator_id, location, content_json, attachments_json)

stage：PRODUCE / PROCESS / INSPECT / WAREHOUSE / LOGISTICS / SALE

content_json：不同环节的表单字段用 JSON 存（毕业设计很省事）

你也可以拆成多张 stage 表，但论文写起来复杂；**“事件表+JSON”**更适合课程设计。

4）核心质量安全明细 + 哈希存证

quality_report(id, batch_id, report_no, agency, result, report_file_url, report_json, created_at)

pesticide_record(id, batch_id, pesticide_name, dosage, usage_date, safe_interval_days, operator, record_json)

hash_notary(id, biz_type, biz_id, sha256, created_at, created_by, remark)

biz_type：QUALITY_REPORT / PESTICIDE_RECORD

biz_id：对应明细表主键

sha256：对“关键字段+文件摘要”的哈希（见后面生成规则）

5）二维码与查询行为（你的创新点）

qr_code(id, batch_id, qr_token, status, created_at, expired_at, remark)

qr_token：加密/签名后的短 token（二维码内容不直接暴露 batch_id）

qr_query_log(id, qr_id, batch_id, query_time, ip, ua, geo_country, geo_province, geo_city, geo_lng, geo_lat, referer)

日志表可做异步写入（MQ 可不用，线程池/Redis 队列也行）

qr_query_stat_day(id, qr_id, day, pv, uv)（可选，用于日报统计与看板）

论文写法：海量过程数据用 MySQL（trace_event），核心质量安全数据用“明细+哈希存证”（hash_notary），实现轻量防篡改。

4. Redis 设计：热点缓存 + 高并发计数（对应研究内容 2.2 / 3.2）
4.1 Key 设计（建议直接写进论文）

批次基础信息缓存

trace:batch:{batchId} → JSON（批次+产品+企业基础信息）

TTL：30min～2h（按你演示需要）

二维码查询计数

trace:qr:pv:{qrId} → Long（总 PV）

trace:qr:uv:{qrId}:{yyyyMMdd} → HyperLogLog（UV 去重，选做）

当天PV临时累计（用于定时落库）

trace:qr:pv:day:{yyyyMMdd} → Hash（field=qrId, value=pv增量）

4.2 计数一致性策略（毕业设计最推荐）

查询接口：每次扫码查询

INCR trace:qr:pv:{qrId}（原子自增）

同时 HINCRBY trace:qr:pv:day:{date} {qrId} 1

落库任务（@Scheduled 每5分钟/每天零点）：

扫描 trace:qr:pv:day:{date} 写入 qr_query_stat_day

查询日志：可选择

方案A（简单）：直接同步写 MySQL qr_query_log（并发低也能撑住）

方案B（更“像样”）：先 LPUSH trace:qr:log，后台线程批量落库（性能更好）

5. “哈希存证”设计（轻量防篡改）
5.1 哈希生成规则（可在论文中描述为“数字指纹”）

对每条核心记录生成 sha256：

取关键字段（例如 batch_id + report_no + agency + result + report_json）

若有附件（PDF/图片），加上文件字节 sha256或文件ETag

拼接成规范化字符串（字段排序固定、JSON 需 canonical 化）

计算 SHA-256 → 写入 hash_notary

5.2 校验方式（消费者/监管可见）

查询展示时，系统重新计算该记录 sha256，与 hash_notary.sha256 对比：

一致：显示“校验通过（未检测到篡改）”

不一致：显示“校验失败（数据疑似被修改）”，并记录审计日志

这样你就实现了“借鉴区块链哈希存证思想，但不引入重链平台”，符合你开题的“务实融合”。

6. 后端接口设计（Spring Boot REST）
6.1 API 分组（建议按 Controller 分包）

/api/auth/**：登录、刷新 token、退出

/api/admin/batch/**：批次 CRUD、关联查询

/api/admin/event/**：节点事件 CRUD（按 stage）

/api/admin/quality/**：质检、用药记录、哈希存证

/api/admin/qr/**：二维码生成/导出/作废/统计

/api/public/trace/**：消费者扫码查询（无需登录）

6.2 关键接口（给你一套可直接实现的清单）
认证

POST /api/auth/login → {username,password} → {token}

GET /api/auth/me → 当前用户信息与菜单权限

批次与节点

POST /api/admin/batch：创建批次（生成 batch_code）

GET /api/admin/batch/{id}：批次详情

GET /api/admin/batch/page：分页筛选（产品、状态、日期）

POST /api/admin/event：新增节点事件（stage + content）

GET /api/admin/event/list?batchId=...：批次全事件（按时间排序）

核心可信数据

POST /api/admin/quality/report：新增质检报告（含上传后的 fileUrl）

服务端：写 quality_report → 计算 sha256 → 写 hash_notary

POST /api/admin/quality/pesticide：新增用药记录 → 同上

GET /api/admin/quality/verify?bizType=...&bizId=...：返回校验结果（也可在查询接口合并返回）

二维码

POST /api/admin/qr/generate：批量生成 {batchId,count,expireDays} → 返回下载链接/或返回 base64 列表（少量时）

POST /api/admin/qr/disable/{qrId}：作废

GET /api/admin/qr/stats?batchId=...：二维码 PV/UV/趋势

公开查询（消费者扫码入口）

GET /api/public/trace/{qrToken}

解析 token → 得到 qrId/batchId（并校验签名、过期、状态）

Redis 自增 PV + 写日志

返回：批次基础信息 + 事件时间轴 + 质检/用药校验状态 + 查询热度

7. 权限模型（Spring Security + JWT）
7.1 角色建议（够毕业设计）

9. 二维码加密/签名方案（毕业设计可落地）
9.1 推荐实现（简单可靠）

二维码内容放：qrToken

qrToken = base64url( qrId + "." + expireAt + "." + HMACSHA256(qrId.expireAt, secret) )

后端查询时：

解析 token

校验 expireAt

重新算 HMAC 对比

查 qr_code 状态是否作废

优点：不用引入复杂加密库；token 不暴露 batchId；可防篡改。

10.2 并发与性能测试（JMeter）

重点测你开题强调的点：高并发扫码查询与计数准确性

场景：100/300/500 并发请求 GET /api/public/trace/{token}

指标：

平均响应时间、95线、吞吐量

MySQL QPS、慢查询

Redis OPS

对比实验：

方案1：不使用 Redis 计数（MySQL update counter）

方案2：Redis INCR + 定时落库

预期结论写法：方案2 响应更快、DB 压力显著下降（用你压测数据佐证）
