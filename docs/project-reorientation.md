# 项目方向调整说明

## 当前项目现状
这个仓库原本是一个前后端分离的农产品追溯系统，已经具备了较完整的领域概念雏形：

- 数据层面已经围绕 `企业 / 产品 / 批次 / 过程记录 / 质检 / 二维码 / 监管 / 反馈` 展开。
- SQL 结构能够支撑批次、追溯记录、质检、二维码和监管留痕。
- 项目被拆分为 `backend`、`admin-web`、`trace-web` 三端，适合做课堂演示和后续扩展。

但也存在明显问题：

- 原工作树中的大量业务源码已经缺失，只剩下壳文件和 SQL、配置、文档残留。
- 原方向更像“功能清单”，模块很多，但主流程表达不够集中。
- 文档表达偏技术堆叠，没有把“为什么要这样设计、谁最需要、先把哪条链路做顺”讲清楚。
- 管理端和公开端页面骨架不完整，无法直接支撑“扫码即看”的核心展示。

## 新的项目定位
项目后续不再追求“功能越多越好”，而是收敛为一个：

**轻量、直观、扫码即看、录入顺手、能演示、能扩展的农产品追溯系统。**

系统价值不在复杂平台能力，而在于用尽量少的步骤完成：

`企业建档 -> 批次创建 -> 过程记录 -> 质检上传 -> 二维码生成 -> 发布 -> 扫码查看`

## 核心使用人
围绕以下 6 类角色设计：

1. 企业管理员
2. 现场录入员 / 生产人员
3. 质检 / 仓储 / 出库相关人员
4. 监管人员
5. 消费者
6. 平台管理员

## 核心使用场景
- 企业管理员快速完成企业建档，并创建可追踪的批次。
- 现场录入员在少字段、少跳转的界面中补录生产过程。
- 质检或仓储人员围绕批次上传质检、补齐仓储和出库留痕。
- 企业在批次完成关键信息后生成二维码并发布。
- 消费者扫码后第一屏就能看到批次摘要、关键时间线、质检情况和异常状态。
- 监管人员可以在异常批次上执行冻结、召回等动作，并保留原因和留痕。

## 主流程
### 1. 企业建档
先完成企业主体、联系人、产地等基础信息。

### 2. 批次创建
以批次为最核心对象，一次建立产品、产地、生产日期与备注。

### 3. 过程记录
按关键节点补录生产、仓储、运输、出库等记录，不追求字段过细。

### 4. 质检上传
补齐支撑发布的质检信息，优先展示检测结论和关键指标。

### 5. 二维码生成与发布
批次具备对外展示价值后再生成二维码、发布公开页面。

### 6. 消费者扫码查看
公开页以“看懂”为目标，不直接暴露后台字段。

## 建议保留模块
- 企业建档
- 产品基础档案
- 批次管理
- 批次过程记录
- 质检上传
- 二维码生成与发布
- 公开追溯查询
- 监管冻结 / 召回 / 留痕

## 建议弱化模块
- 复杂审批流
- 重消息中心
- 过多统计大屏
- 过细权限拆分
- 与主流程弱关联的独立管理页
- 为了“展示功能多”而存在的辅助模块

## 与新方向冲突的地方
- 原项目表达上更偏“模块罗列”，不够突出批次中心。
- 公开追溯页的价值在旧说明里不够靠前。
- 数据库中已有批次主线，但前端和文档没有把它提到系统中心。
- Docker 与文档里存在硬编码密码、历史默认值，不利于规范展示。
- 当前仓库源码缺失，导致“能跑通的主链路”与“文档叙述的能力范围”不一致。

## 本轮改造结论
本轮已经完成第一轮方向收敛：

- 后端改为围绕批次的统一接口骨架。
- 管理端收敛为总览、批次列表、批次工作台三类页面。
- 公开端围绕扫码追溯首屏重组展示结构。
- 状态流转收敛为 `DRAFT / PUBLISHED / FROZEN / RECALLED`。
- 公开查询返回结构重构为“首屏摘要 + 时间线 + 质检 + 企业 + 异常提示”。
