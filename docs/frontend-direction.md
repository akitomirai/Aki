# 前端页面改造方向

## 1. 企业后台应该保留哪些页面
后台优先保留与主流程强相关的页面：

- 总览页
- 批次列表页
- 批次工作台页
- 企业建档页
- 产品基础档案页

如果后续继续扩展，再补：

- 监管异常处理页
- 平台企业审核页

## 2. 哪些页面以“批次”为中心重组
以下能力不再拆成孤立模块页，而应尽量并入批次工作台：

- 过程记录
- 质检信息
- 二维码状态
- 发布状态
- 冻结 / 召回动作

也就是说，用户进入一个批次后，应尽量在同一页完成“看信息 + 做动作”。

## 3. 公开追溯页首屏应该展示什么
首屏必须优先展示消费者能马上理解的信息：

- 产品名称
- 批次号
- 企业名称
- 产地
- 当前状态
- 最近质检结论
- 是否存在冻结 / 召回等异常

不要把后台内部字段、复杂 JSON、无解释的代码值直接抛给消费者。

## 4. 哪些交互要优先优化
- 批次创建时减少字段数量，先完成最小建档。
- 过程记录支持“按节点快速补录”。
- 批次详情页少跳转，尽量在一个工作台内聚合。
- 二维码生成与发布动作尽量一眼可见。
- 公开追溯页首屏不需要滚动太久就能看懂。

## 5. 哪些功能暂不做重
- 复杂大屏
- 复杂审批流
- 多层级消息中心
- 过细的角色菜单切换
- 与主流程无强关联的专题看板

## 本轮已落地的最小页面调整
### admin-web
- `DashboardView`：突出主流程、建设重点和批次概览
- `BatchListView`：把批次管理作为主入口
- `BatchWorkbenchView`：聚合基本信息、状态、记录、质检、二维码和动作入口

### trace-web
- `TraceDetailView`：重构为面向消费者的公开追溯页，强调摘要、时间线、质检和异常提醒
