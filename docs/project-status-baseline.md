# 项目现状基线与后续待办清单 (Project Status Baseline & TODO)

本文件旨在记录农产品质量安全溯源系统的当前技术状态、功能覆盖情况及后续开发优先级，作为项目推进的统一指引。

## 1. 项目现状总览
目前项目已建立起稳健的后端架构与管理端原型，整体处于**“核心功能已闭环，部分模块待增强，部署体系待建立”**的阶段。

- **后端 (backend)**: 架构高度规范，模块化程度高（14 个业务模块），已完成权限分层 Controller 设计。
- **管理端 (admin-web)**: 基于企业角色的管理功能已初具规模，UI 框架与状态管理（Pinia）已打通。
- **移动端 (trace-web)**: 处于原型阶段，具备基础溯源展示，但交互深度与 C 端功能缺失较多。
- **部署层 (docker)**: 仅有 SQL 初始化脚本，属于“预留状态”。

## 2. 子项目深度现状

### 2.1 admin-web (后台管理前端)
- **定位**: 企业端（Enterprise）与监管/平台端的管理门户。
- **关键目录**: `src/pages` (业务页面), `src/composables` (业务逻辑复用), `src/api` (接口封装)。
- **已完整部分**: 
  - 登录/注册流程。
  - 生产批次管理、详情查看、二维码生成预览。
  - 质检报告上传与管理。
  - 仪表盘（Dashboard）基础统计。
- **待补全部分**: 
  - **监管端/平台端页面**: 当前 UI 侧重于企业端，针对 `RegulatorOrgPlatformController` 和 `SysUserPlatformController` 的 UI 尚未完全展开。
  - **反馈处理**: `FeedbackEnterpriseController` 对应的管理界面需完善。
- **当前判断**: 适合进行业务功能细化开发，UI 结构稳定。

### 2.2 backend (后端服务)
- **定位**: 系统的核心大脑，提供 API 支持。
- **controller 分层现状**:
  - `auth`: 认证与授权。
  - `enterprise`: 企业侧业务（批次、溯源、质检、反馈）。
  - `platform`: 平台侧管理（公司、用户、监管机构、邀请码）。
  - `regulator`: 监管侧记录（监管操作）。
  - `publicapi`: C 端匿名查询。
- **已完整模块**: `auth`, `batch`, `product`, `qr`, `company`, `log` (含脱敏规范)。
- **待验证/增强模块**: 
  - **notary (公证存证)**: 存证快照逻辑 `NotarySnapshotBuilder` 需结合真实业务数据验证。
  - **regulation (监管记录)**: 监管流程的触发与状态流转需与前端进一步联调。
- **当前判断**: 架构成熟，适合作为基石支持全链路联调。

### 2.3 trace-web (移动端溯源前端)
- **定位**: 消费者扫描产品二维码后的信息展示窗口。
- **关键目录**: `src/pages/TraceDetail` (核心溯源流)。
- **已完整部分**: 基础的溯源时间轴（Timeline）展示、样式基础构建。
- **待补全部分**: 
  - **消费者反馈**: 尚未对接 `FeedbackPublicController`。
  - **扫码日志记录**: 需配合 `QrPublicController` 完善扫码后的后端统计逻辑。
  - **UI/UX 优化**: 需根据移动端适配 Vant 组件，增强交互感。
- **当前判断**: **半成品状态**，是目前全链路中的薄弱环节。

### 2.4 docker / docs (配套资源)
- **docker**: 处于“预留”状态，急需补齐 `Dockerfile` 与 `docker-compose.yml`。
- **docs**: 已建立索引，但内容厚度不足。

## 3. 后续开发优先级 (Priority)

### P0：核心链路闭环与环境标准化
1. **容器化落地**: 在 `docker/` 下补齐 `docker-compose.yml` 及各模块 `Dockerfile`，确保开发环境一致性。
2. **移动端溯源增强**: 补齐 `trace-web` 的反馈提交功能，并对接后端扫码统计 API。
3. **全链路联调测试**: 从“企业创建批次 -> 监管审核 -> 生成二维码 -> 移动端扫码 -> 存证快照”进行完整闭环跑通。

### P1：管理端功能下沉与深度文档
1. **管理端多角色适配**: 补齐平台管理员与监管机构所需的管理页面。
2. **公证存证逻辑落地**: 完善 `notary` 模块，确保关键业务数据变更时自动触发存证记录。
3. **API 文档精修**: 基于 Knife4j 完善 OpenAPI 定义，并导出离线开发指南。

### P2：系统加固与体验优化
1. **日志展示控制台**: 在 `admin-web` 中增加审计日志查看页面。
2. **性能与监控**: 优化 ECharts 大数据展示，引入后端监控指标。
3. **自动化测试**: 编写核心业务（如批次状态变更）的 JUnit 测试用例。

## 4. 推荐协作说明
开发者应优先阅读 `docs/README.md` 建立文档感知，随后按照 **P0** 任务逐步推进。
