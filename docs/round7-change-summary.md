# Round 7 Change Summary

## 本轮目标

本轮只围绕四类真实使用场景做体验打磨，不扩新业务模块：

- 企业管理员主流程
- 现场快速补录
- 消费者扫码查看
- 监管 / 答辩演示

核心标准是：少步骤、少录入、少跳转、少理解成本，并继续以“批次”作为主工作对象。

## 本轮主要改动

### 1. 企业管理员主流程

- 重做 `admin-web` 批次列表页表达，让它更像后台主入口而不是普通列表。
- 默认增加“待处理 / 可发布 / 风险批次 / 已发布 / 全部批次”视角，帮助管理员直接聚焦当前最该处理的批次。
- 每个批次卡片增加“下一步推荐”、进度标签和缺失项提示，减少用户自己判断该做什么的成本。
- 批次创建完成后直接跳转到工作台，并通过引导条提示后续推荐顺序：补追溯、上传质检、生成二维码、发布批次。

### 2. 批次工作台体验

- 将工作台顶部改造成任务型摘要区，突出当前状态、下一步推荐和完成检查项。
- 快捷操作改为按业务场景分组：补追溯、上传质检、生成二维码、状态处理。
- 风险区补充“最近处理动作”和“恢复发布前检查项”，让监管和答辩展示时更容易讲清楚异常原因、整改动作和恢复条件。

### 3. 现场快速补录

- 追溯补录继续沿用弹层，不改成深层跳转。
- 增加阶段快捷选择、常用地点模板、说明模板、复制上一条并微调、保存并继续。
- 上传图片后的反馈改成更贴近用户语言，成功后会立即显示在当前记录区域。

### 4. 消费者扫码查看

- 重排 `trace-web` 公开页首屏，优先展示产品名、当前状态、质检结论、企业名、批次号和产地。
- 用更偏消费者视角的文案表达当前结论，减少后台字段感。
- 时间线默认先展示关键节点，避免用户一上来看到过长内容，同时保留“查看更多关键节点”入口。
- 风险状态改成普通用户更容易理解的表达，例如“当前已暂停流通”“当前已召回”“当前正在处理中”。

### 5. 演示运行支持

- 将 `backend` 中的 H2 依赖从仅测试可见调整为 `runtime` 可用，方便本地使用测试配置承载答辩 / 演示数据。
- 新增 `demo` profile，启动 backend 后可直接加载一组稳定可展示的企业、产品、批次、风险样例。
- 新增一键启动脚本与预检脚本，减少答辩当天手工拼环境的成本。

## 为什么这样改

- 批次列表页是管理员最常进入的页面之一，必须一眼看出哪些批次缺什么、下一步做什么。
- 工作台是整个系统最适合答辩和业务演示的页面，应该像业务中心，而不是信息拼盘。
- 现场补录最怕重复输入和跳转负担，所以重点做复用、模板和连续录入。
- 公开页的目标不是展示后台字段，而是让消费者先快速得到结论，再决定是否继续看过程。

## 涉及范围

- `admin-web`
- `trace-web`
- `docs`
- `scripts`
- `backend demo profile`

后端本轮原则上只复用现有字段和动作能力，没有额外扩复杂模块。

## 验证计划

本轮按以下顺序验证：

1. `backend` 执行 `mvn compile` 与 `mvn test`
2. `admin-web` 执行 `npm run build`
3. `trace-web` 执行 `npm run build`
4. 如本地服务能顺利启动，再补一次完整链路联调验证

## 稳定演示样例

- `DEMO-ORANGE-202603-A1`：正常发布批次，适合扫码展示
- `DEMO-ORANGE-202603-D1`：草稿批次，适合管理员主流程演示
- `DEMO-TEA-202603-F1`：冻结批次，适合风险处理中与恢复前检查项演示
- `DEMO-RICE-202603-R1`：召回批次，适合监管 / 答辩演示

## 备注

如果构建或联调中发现本轮页面改造需要最小后端辅助字段，再仅补轻量字段，不扩大后端范围。
