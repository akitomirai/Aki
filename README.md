# 农产品追溯系统

## 1. 项目简介
这是一个面向农产品追溯场景的轻量化系统，包含后端服务、企业管理端和公开追溯页三部分。它不追求做成“大而全的平台”，而是聚焦一条真正能跑顺、能演示、能被用户理解的追溯主链路。

## 2. 项目定位
项目定位是：

**轻量、直观、扫码即看、录入顺手、能演示、能扩展的农产品追溯系统。**

系统的业务中心不是零散模块，而是“批次”；系统最重要的价值出口不是后台，而是“二维码公开追溯页”。

## 3. 项目目标
- 让企业用尽量少的步骤完成追溯建档
- 让录入员在现场也能快速补录关键过程信息
- 让质检、仓储、出库等信息围绕同一个批次聚合
- 让监管动作能落在批次状态上并保留原因
- 让消费者扫码后一眼看懂这批产品的来龙去脉

## 4. 适用人群与使用工具
适用角色包括：

- 企业管理员
- 现场录入员 / 生产人员
- 质检 / 仓储 / 出库相关人员
- 监管人员
- 消费者
- 平台管理员

本仓库提供三端支撑：

- `backend`：Spring Boot 后端接口
- `admin-web`：企业 / 平台使用的管理端
- `trace-web`：消费者扫码查看的公开追溯页

## 5. 真实使用场景
一个企业要展示某批赣南脐橙的来源和过程，不需要先进入很多管理模块。更理想的方式是：

企业先建档，创建批次，再按关键节点补录几条过程记录，上传一份质检结果，生成二维码并发布。随后消费者扫描包装上的二维码，就能看到产地、关键时间线、质检信息和异常状态。

这就是本项目现在聚焦的核心场景。

## 6. 核心业务流程
系统优先保证以下主流程顺畅：

`企业建档 -> 批次创建 -> 过程记录 -> 质检上传 -> 二维码生成 -> 发布 -> 扫码查看`

这条链路同时服务演示、课堂答辩和后续真实扩展。

## 7. 功能模块设计
当前建议保留的核心模块如下：

- 企业建档
- 产品基础档案
- 批次管理
- 批次过程记录
- 质检信息
- 二维码生成与发布
- 公开追溯查询
- 冻结 / 召回等状态处理

这些模块不是彼此平铺，而是尽量收拢到“一个批次”的视角下组织。

## 8. 页面设计思路
### 管理端
管理端优先保留三类高价值页面：

- 总览页
- 批次列表页
- 批次工作台页

其中“批次工作台”是管理端核心，它应该聚合：

- 批次基本信息
- 当前状态
- 过程记录
- 质检状态
- 二维码状态
- 可执行动作入口

### 公开端
公开追溯页首屏优先展示：

- 产品名
- 批次号
- 企业名
- 产地
- 当前状态
- 最近质检结论
- 是否有冻结 / 召回异常

## 9. 交互优化原则
所有交互调整都围绕这几个原则：

- 少录入
- 少跳转
- 少理解成本
- 让用户顺着批次往下走
- 让消费者一眼看懂

这意味着系统不适合继续堆复杂流程、复杂审批或大量独立页面。

## 10. 技术方案
### 后端
- Spring Boot 3
- Spring Security
- MyBatis-Plus 依赖保留，作为后续数据库接入基础
- 统一返回结构 `ApiResponse`

### 前端
- Vue 3
- Vite
- Element Plus
- Axios

### 数据层
- 当前保留 `sql/` 中的数据库结构与种子数据，作为后续持久化接入基础
- 本轮为了保证仓库可运行，后端先提供基于内存演示数据的批次主链路

## 11. 当前建设重点
本轮优先建设的不是更多模块，而是：

- 把“批次”做成系统中心
- 把“公开追溯页”做成核心价值出口
- 把“批次详情”做成工作台
- 把状态流转收敛到 `DRAFT / PUBLISHED / FROZEN / RECALLED`
- 把接口返回结构重组得更适合前端和消费者理解

## 12. 不作为当前重点的方向
以下方向并不是当前重点：

- 复杂审批流
- 过重消息中心
- 大屏展示优先化
- 过细权限拆分
- 过于复杂的统计报表
- 为了显得功能多而增加的边缘模块

## 13. 项目价值
这个项目的价值主要体现在三点：

- 能把追溯能力讲清楚，而不是只展示一堆表单
- 能快速演示“从后台录入到消费者扫码”的闭环
- 能在后续逐步接数据库、接真实上传、接角色权限，而不推倒重来

## 14. 项目愿景
理想状态下，这个系统应当成为一个“把复杂追溯问题尽量做轻”的样板：

企业愿意用，录入员不嫌麻烦，消费者能看懂，监管人员也有抓手。

## 15. 后续计划
短期内建议继续推进：

1. 企业建档页与批次创建表单
2. 批次工作台内的真实录入动作
3. 质检附件与二维码发布联动
4. MySQL + MyBatis-Plus 持久化接入
5. 监管异常处理页

## 16. 开发与运行说明
### 环境建议
- JDK 17+
- Maven 3.8+
- Node.js 20+

### 后端
如果本机 `JAVA_HOME` 未配置正确，可临时指定 JDK 路径再运行：

```powershell
$env:JAVA_HOME='C:\Program Files\Java\jdk-21'
$env:Path="C:\Program Files\Java\jdk-21\bin;$env:Path"
cd backend
mvn spring-boot:run
```

默认地址：

- 后端接口：[http://127.0.0.1:8080](http://127.0.0.1:8080)
- Swagger：[http://127.0.0.1:8080/swagger-ui.html](http://127.0.0.1:8080/swagger-ui.html)

### 管理端
```bash
cd admin-web
npm install
npm run dev
```

访问地址：[http://127.0.0.1:5174](http://127.0.0.1:5174)

### 公开追溯端
```bash
cd trace-web
npm install
npm run dev
```

访问地址：[http://127.0.0.1:5173/t/test-token-2026](http://127.0.0.1:5173/t/test-token-2026)

### 数据库脚本
当前仓库保留了数据库结构与种子数据，位于：

- `sql/01_schema.sql`
- `sql/02_seed_dev.sql`
- `sql/03_reset_dev.sql`

本轮后端为了保证仓库可运行，先使用内存演示数据；后续可按这些 SQL 接回 MySQL。

## 17. 推荐演进方向
推荐按以下顺序演进，而不是同时摊大：

1. 先把企业建档、批次创建、工作台操作做完整
2. 再把内存演示数据替换成数据库持久化
3. 再补附件上传、监管页和权限
4. 最后再做统计、看板和更细的角色协作

## 18. 总结
这个项目现在的方向已经明确：

不是做功能堆砌的后台集合，而是做一个围绕批次展开、以扫码追溯页为最终价值出口的农产品追溯系统。

如果你希望它适合毕业设计展示，这条路线更清楚；如果你希望它后续继续演进成真实系统，这条路线也更稳。
